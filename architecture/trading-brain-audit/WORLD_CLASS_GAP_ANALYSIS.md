# World-Class Gap Analysis

Date: 2026-07-17

Scope: forensic analysis only. This document compares the current trading brain against the capabilities normally expected of a highly disciplined institutional or professional systematic trading platform. It does not recommend relaxing guardrails or changing production behaviour in this sprint.

## Executive Finding

AI Trader is stronger as a governed execution and audit platform than it is as a mature trading intelligence platform.

The codebase has meaningful engineering strengths:

- broker separation through adapter classes in `src/ai_trader/broker_adapters.py`;
- deterministic risk and policy validation through `src/ai_trader/guardrails.py`, `src/ai_trader/foundation.py`, and `src/ai_trader/orchestrator.py`;
- persistent recommendation and trade audit records through the audit database paths used by `src/ai_trader/api.py`;
- broker polling, exit monitoring, auto execution workers, and push dispatch workers in `LocalApiService.start_background_workers`;
- read-only founder explanation through `OpenAIReadOnlyExplainer` in `src/ai_trader/ai.py`.

The trading brain is not yet world-class because it lacks formal strategy definitions, empirical validation, calibrated confidence, regime-aware signal selection, canonical lifecycle attribution, and adaptive but governed learning loops.

## Capability Scorecard

| Capability | Current maturity | Evidence | World-class expectation |
|---|---:|---|---|
| Governance and guardrails | 3/5 | `validate_trade_proposal`, `InvestmentOrchestrator.evaluate_recommendation`, Kraken live-order checks | Strong controls with policy versioning, simulation, and formal risk sign-off |
| Broker abstraction | 3/5 | Alpaca and Kraken adapters; future broker sections in UI/API | Unified broker contract, capability registry, broker-specific cost/routing model |
| Recommendation persistence | 3/5 | `agent_proposal` events stored and retrieved by recommendation endpoints | Canonical recommendation lifecycle with strategy, signal, regime, expiry, and outcome |
| Strategy intelligence | 1/5 | No strategy registry; crypto uses positive trend heuristic; equity relies on OpenAI or demo probe | Named strategies with explicit entries, exits, filters, invalidations, expected R, and backtests |
| Market regime awareness | 0/5 | No regime classifier found | Regime engine that blocks or selects strategies based on trend, volatility, liquidity, and risk-on/off |
| Signal quality | 2/5 | Crypto scores trend/momentum/liquidity/risk from public market metrics; many fields are stored but unused | Multi-timeframe, volume, volatility, spread, trend, catalyst, and relative strength signals with provenance |
| Confidence calibration | 1/5 | Confidence is heuristic; threshold such as 85% is not empirically calibrated | Confidence buckets tied to observed win rate, expectancy, drawdown, sample size, and strategy/regime |
| Quantitative validation | 1/5 | Unit tests pass; no backtest or out-of-sample validation found | Backtesting, walk-forward validation, paper/live comparison, slippage/fee modelling |
| Learning | 1-2/5 | Reports and learning notes exist; outcomes do not modify future strategy parameters | Outcome attribution feeds reviewed model improvements, calibration, and founder-approved strategy changes |
| Trade lifecycle | 2/5 | Raw broker rows and some managed-exit attribution exist | Canonical lifecycle from recommendation to entry, partial fills, stop/target updates, exit, P&L, fees |
| Explainability | 3/5 | Ask UI and reports explain from stored evidence | Evidence-linked explanations with confidence provenance and exact reason codes |
| Operational resilience | 3/5 | Background workers and tests cover many failure paths | Observability dashboard, alerting, replay, idempotency audit, reconciliation guarantees |

## Gap 1: No Formal Strategy Layer

Current implementation:

- Equities are proposed by `AITradingAgent.propose_trades` in `src/ai_trader/agent.py`.
- If `OPENAI_API_KEY` is configured, `OpenAIProposalAnalyzer.propose` in `src/ai_trader/ai.py` asks OpenAI to return a conservative JSON trade proposal.
- If OpenAI is not available, production equities mostly produce no trade through `_no_trade_probe`; demo mode can emit a validation proposal.
- Kraken crypto proposals are generated by `propose_crypto_trades` using `CRYPTO_RESEARCH_SCORES`, minimum confidence, positive trend, fixed stop, and fixed 2R target.

Missing:

- named strategy IDs;
- explicit timeframe per strategy;
- entry setup definition;
- invalidation condition;
- allowed market regimes;
- expected holding period;
- expected R multiple;
- historical performance by strategy;
- strategy approval status.

Impact:

The app can propose and validate trades, but it cannot yet say, in a rigorous way, "this trade is a valid instance of Strategy X, which historically performs well in Regime Y."

## Gap 2: Confidence Is Not Probability

Current implementation:

- Crypto confidence is normally the `overall_due_diligence_score` stored in `CRYPTO_RESEARCH_SCORES`.
- Bootstrap crypto scoring can seed allowed Kraken pairs at or above the configured minimum confidence.
- Equity confidence may be produced by OpenAI or by demo logic.
- `calculate_investment_score` in `src/ai_trader/foundation.py` blends available summaries and a risk score into a composite.

Missing:

- calibration of 85% confidence against realized outcomes;
- confidence bucket performance;
- separate confidence for asset quality, setup quality, execution quality, and exit quality;
- sample-size controls;
- downgrade rules when live or paper results contradict high confidence.

Impact:

An 85% XRP recommendation and a 90% equity recommendation are not currently proven to be comparable, nor are they proven to mean "85% chance of profit."

## Gap 3: No Market Regime Engine

Current implementation:

- Crypto has trend, momentum, volatility, liquidity, and risk scores.
- `propose_crypto_trades` checks that technical trend score is above 0.5.
- The scheduler refreshes research periodically.

Missing:

- trending versus ranging classification;
- high-volatility versus low-volatility state;
- risk-on versus risk-off state;
- liquidity stress detection;
- regime-specific strategy blocking;
- regime-specific sizing.

Impact:

The system may treat a volatile selloff, a calm uptrend, and a choppy sideways market too similarly.

## Gap 4: Investment Quality And Trade Setup Are Blended

Current implementation:

- `calculate_investment_score` blends fundamentals, technicals, market, macro, behavioural, policy fit, and risk.
- Recommendations display a combined confidence and investment score.

Missing:

- asset quality score;
- short-term setup score;
- execution feasibility score;
- risk/reward score;
- catalyst score;
- regime fit score.

Impact:

A strong company or strong coin can appear attractive even when the immediate trade setup is weak, and a technically attractive setup can be confused with a good long-term investment.

## Gap 5: Trade Learning Is Mostly Reporting

Current implementation:

- The system records trade events, broker history, orchestrator decisions, performance attribution, reports, and benchmark learning notes.
- `daily_learning_update` produces lessons and recommendations.
- The Ask screen can explain from stored evidence and, when available, OpenAI can convert evidence into plain English.

Missing:

- automatic strategy performance attribution by strategy ID;
- signal-level outcome attribution;
- closed-trade datasets with clean entry/exit/fees/slippage;
- confidence recalibration from completed trades;
- founder-approved promotion/demotion of strategies.

Impact:

AI Trader accumulates evidence, but the trading technique itself does not yet improve automatically from that evidence.

## Gap 6: Broker Rows Are Not Enough For Learning

Current implementation:

- Alpaca and Kraken broker rows are polled and stored.
- The Trade History UI can display broker fills and some open/closed state.
- Managed exits can record closed trade attribution when enough entry and exit information exists.

Missing:

- canonical trade lifecycle table;
- deterministic matching from recommendation to order to fills to position to exit;
- partial-fill normalization;
- fees and slippage as first-class fields;
- live unrealized P&L per open AI-managed trade.

Impact:

Reports and Ask answers can still show "not available" where the broker data exists but has not been normalized into an analysis-ready trade lifecycle.

## Gap 7: No Quantitative Proof Of Edge

Current implementation:

- Unit tests validate software behavior.
- Paper/live broker records exist.
- Guardrails protect capital and workflow.

Missing:

- backtesting;
- walk-forward testing;
- out-of-sample testing;
- slippage and fee modelling;
- benchmark comparison;
- drawdown distribution;
- Monte Carlo or bootstrapped risk analysis;
- strategy retirement criteria.

Impact:

The app may be operationally ready to place controlled trades, but it has not yet proven a durable statistical edge.

## Gap 8: Data Sources Are Too Thin For Advanced Trading

Current implementation:

- Alpaca provides account, positions, orders, fills, news, latest bars, clock, and asset availability.
- Kraken provides balances, orders, trade history, and current prices.
- CoinGecko provides crypto market snapshots and basic price/volume/cap movements.
- OpenAI can generate proposals or explanations where configured.

Missing:

- reliable historical candles for backtesting across all traded instruments;
- order book and spread data;
- robust news/sentiment provider for crypto;
- on-chain data provider;
- economic/calendar event feeds;
- corporate earnings/catalyst data;
- data quality scoring.

Impact:

The current market picture is sufficient for basic recommendations, but not yet sufficient for advanced execution timing or professional-grade opportunity selection.

## What Is Already Worth Protecting

The following should be protected while improving the trading brain:

- Orchestrator authority: no broker should be called directly by AI output.
- Independent broker controls and broker-specific seatbelts.
- Environment-variable gates for live order submission.
- Audit-first design.
- Founder read-only Ask UI.
- Recommendation freshness and expiry.
- Manual and automatic execution paths that both pass through validation.
- Existing test suite as the baseline safety net.

## World-Class Target State

A mature version of AI Trader should be able to answer:

1. What exact strategy generated this trade?
2. What regime was present?
3. What signal evidence supported entry?
4. What was the expected reward-to-risk?
5. What did similar historical trades do?
6. What confidence bucket does this belong to, and how has that bucket performed?
7. What caused exit?
8. Did the trade follow the plan?
9. Did the result improve or weaken trust in the strategy?
10. What change is recommended for founder approval?

The current implementation can answer some operational and risk questions, but not all of these trading-intelligence questions yet.
